# 📋 “系统说明书-Issue”方案（SIP）标准实施白皮书

## 一、 方案背景与核心痛点

在企业级私有内网环境的软件研发中，将传统的“需求功能点估算表”直接转化为“可运行代码”存在巨大的技术与管理断层。主要面临以下三大核心痛点：

1. **信息熵极高（模糊瞎写）：** 原始功能点描述（如电力系统中的“台账比对”、“流程同步”）字数极少、业务边界模糊。若直接交付大模型编码，大模型会严重“脑补”技术方案，导致产出垃圾代码。
    
2. **内网物理隔离限制：** 研发环境处于纯内网，无法直连公网 AI 服务，但内网本身具备极其庞大且完善的基础设施（GitLab/Jira、Maven/NPM 私服、Nacos、配置中心等）。
    
3. **上下文爆炸与交叉污染：** 复杂的跨级业务（省-地-县）逻辑交织。若让 AI 协同全量上下文，会导致大模型 Context 瞬间耗尽，且不同模块间极易产生行级语义的交叉污染。
    

**方案核心哲学：** 引入“系统说明书”**与**“开发 Issue”作为两道物理减熵屏障。**需求说清楚（靠说明书收敛边界），代码写明白（靠 Issue 隔离上下文）。**

## 二、 全栈工具链选择（Tooling Selection）

基于完全离线的物理限制，方案放弃了重型的端到端沙箱框架，选择深度复用内网已有资产，组装轻量、高可控的开源/本地工具链：

```
+------------------------------------------------------------------------+
| 阶段一：业务减熵（Dify Workflow / 本地 Markdown FSM）                 |
+------------------------------------------------------------------------+
                                   │ (生成：结构化说明书 + ADR)
                                   ▼
+------------------------------------------------------------------------+
| 阶段二：任务切碎（LangGraph / 内网 GitLab REST API）                     |
+------------------------------------------------------------------------+
                                   │ (生成：行级原子化开发 Issues)
                                   ▼
+------------------------------------------------------------------------+
| 阶段三：精准编码（Claude Code / 挂载 GitLab MCP Server）                |
+------------------------------------------------------------------------+
```

- **业务减熵层：Dify（开源自托管版）**
    
    - _选型理由：_ 支持完全离线 Docker 部署。利用其强大的 Workflow 节点与表单交互能力，作为扮演业务分析师（BA）的载体，通过界面对人类发起追问，卡死模糊需求。
        
- **任务编排层：LangGraph + GitLab / Jira API**
    
    - _选型理由：_ 利用有向无环图（DAG）的严谨状态控制，将通过评审的说明书，严格按照预设的 JSON Schema 自动切碎，通过 REST API 灌入内网 GitLab/Jira 变成物理 Issue。
        
- **精准执行层：Claude Code + GitLab MCP Server**
    
    - _选型理由：_ 研发人员主力工具。通过离线挂载 GitLab MCP 插件，赋予 Claude Code 联动内网 Issue 的能力，使其保持极小的上下文，专职攻克原子任务。
        

## 三、 标准标准流程（Pipeline Framework）

本方案包含**三大标准阶段、六个核心动作**，严格遵循 **Maker-Checker（四眼原则）** 安全红线。

### 阶段一：业务减熵（需求说清楚）

- **动作 1：需求透传与模糊度扫描 (Ingestion & Scan)**
    
    - 系统导入二级需求表，AI 逐行扫描。识别出技术通道不明确、策略缺失、权限边界模糊的节点，触发状态机将该行标记为 `NEED_CLARIFY`。
        
- **动作 2：苏格拉底式多轮澄清 (Multi-turn Clarification)**
    
    - AI 限制单次提问数量（3-5 个），通过结构化表单或对话向人类（架构师/PO）发问。人类补全关键抉择（例如：明确指出台账比对采用“定时任务+REST接口”）。
        
- **动作 3：固化系统说明书与 ADR (Specification Freezing)**
    
    - AI 结合人类答复，自动生成包含 **ADR（架构决策记录）**、**Gherkin（BDR 行为边界）** 和 **ILF/EIF 数据模型** 的标准 Markdown 说明书。人类一键确认后，状态变更为 `APPROVED`。
        

### 阶段二：任务切碎（中转为 Issue）

- **动作 4：结构化原子Token化 (Tokenization & Mapping)**
    
    - 编排引擎读取 `APPROVED` 的说明书，启动多智能体（BA、QA、架构智能体）进行红蓝对抗评审，确保无逻辑漏洞后，按照“数据库 $\rightarrow$ 后端算法 $\rightarrow$ 接口 API $\rightarrow$ 前端渲染”的依赖链条切碎需求。
        
- **动作 5：内网研发生态灌入 (Issue Injection)**
    
    - 通过内网 API 自动在 GitLab 创建 Issue 任务链。Issue 评论区天然承载该任务的独立上下文与验收标准（Acceptance Criteria）。
        

### 阶段三：精准编码（代码写明白）

- **动作 6：Claude Code 挂载执行 (Atomic Coding)**
    
    - 开发人员在本地激活 Claude Code，通过 MCP 协议拉取指定 `Issue ID`。
        
    - Claude Code 在极小的上下文空间内，仅阅读该 Issue 对应的逻辑，调用内网 Maven/NPM 私服进行本地编译与单元测试，通过后自动 `git commit -m "feat: close #id"` 提交，进入内网持续集成（CI）流水线。
        

## 四、 说明书完整性控制（Quality Gates）

为防止“系统说明书”流于形式，SIP 方案在阶段一和阶段二之间强制设立**三大自动化质量红线（Linting）**：

1. **Schema 强校验：** 说明书必须采用元数据（Front Matter）声明状态，必须包含 `## 核心数据模型`、`## 安全与权限边界` 章节，否则编译报错，无法进入 Issue 拆解阶段。
    
2. **ADR 唯一性锁定：** 凡是涉及多种实现方案的通用功能（如登录、同步、纠错），说明书中必须存在 `ADR-XXX` 记录，且状态必须为人类显式确认的 `[CONFIRMED_BY_HUMAN]`。
    
3. **红蓝对抗（Red Teaming）：** 引入专门的“找茬 Agent”，拿着内网高可用和安全合规检查单（如：密码防爆破、网闸摆渡超时熔断）对说明书进行自动化拷问。
    

## 五、 方案优势总结

- **零公网依赖：** 全套流程流转于 Dify、GitLab、Markdown 文件与 Claude Code 之间，完美适配庞大的隔离内网。
    
- **终结幻觉与瞎写：** AI 的编码冲动被死死锁在“说明书多轮确认”和“Issue 原子原子化”的两道大闸之内。
    
- **极致性能：** 避免了全量代码分析导致的 Context 膨胀，通过 Issue 隔离，每次 Tool Call 耗时维持在秒级，极大节约内网算力集群的吞吐量（Throughput）。
