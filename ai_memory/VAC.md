在**解除内网物理隔离限制、全量拥抱公网 SaaS 级顶尖大模型、弹性算力集群及 AI 原生代研发工具链**的背景下，原有的“系统说明书-Issue”方案将迎来真正的终极完全体形态。

这套方案正式命名为：**“全栈视觉化原型与云端 Agent 自动闭环流水线”（Visual Prototype to Agentic Closure Pipeline, 简称 VAC 方案）**。

以下是 VAC 方案的完整实施白皮书。

# 🚀 “视觉化原型与云端 Agent 自动闭环流水线”方案（VAC）标准实施白皮书

### 暨 全公网 SaaS 生态下 AI 原生级（Agent-First）端到端研发无缝闭环方案

## 1. 方案概述与演进哲学

在完全连接公网、拥有全球顶级大模型（如 Claude 3.5 Sonnet 官方 API、GPT-4o）以及弹性高并发算力的环境下，软件研发的生产力范式发生了解构与重组。

### 1.1 核心痛点演进

- **内网 SIP 方案的局限：** 原内网 SIP V2.0 方案虽然通过 Issue 拦截了 AI 瞎写的冲动，但人类依然需要面对抽象的 Markdown 说明书，无法直观感知页面交互，且人类在 GitLab 和本地 IDE 之间频繁切换，流程仍有明显的工程割裂。
    
- **信息密度的跨越式跃迁：** 模糊的二级需求表向具体代码的转化，最核心的障碍是“视觉与交互的模糊”。只要前端界面和数据流向在第一阶段能够变成“看得见、可交互”的 Live Preview（实时预览），后续的代码生成、单元测试与云端 CI/CD 就能实现近乎 100% 的无人值守（Automated）交付。
    

### 1.2 VAC 方案哲学

VAC 方案的核心哲学是：**“人类把控业务方向、AI 视觉原型对齐、云端 Agent 自主编码测试”**。

- **需求说清楚（视觉化对齐）：** 将模糊表格直接转化为可交互的前端 Live 原型，人类在网页上像改图纸一样“指点”AI 修正需求，实现物理世界的彻底减熵。
    
- **代码写明白（全自动化闭环）：** 云端 Agent（如 GitHub Copilot Workspace）直接认领任务，自主切分支、自主编码、自主运行云端沙箱测试，人类只需在浏览器上点一下 `Merge`，代码直接上线。
    

## 2. 工具链选择（Tooling Selection）

解除内网限制后，方案彻底放弃自组装脚本，全面拥抱目前硅谷最前沿的 **AI 原生代研发生态（AI-Native SaaS）**：

- **业务减熵与视觉对齐层：Lovable.dev / V0.dev**
    
    - _选型理由：_ 允许直接导入复杂的需求表格。利用顶尖公网模型的全量推理能力，不仅能读懂电力调度的复杂规则，还会**直接生成一个包含真实数据 Mock、可点击跳转、带高亮效果的前端系统原型（Live App）**。
        
- **敏捷任务承载层：Linear (SaaS版)**
    
    - _选型理由：_ 全球公认研发体验最好的现代化项目管理平台。其高度集成的 GraphQL API 与 Webhook，能够完美与 AI 原生工具互通，自动生成带有完美依赖关系的 Epic、Stories 和 Tasks。
        
- **云端自主执行层：GitHub Copilot Workspace / Cursor (Agent Mode)**
    
    - _选型理由：_ 真正的云端一体化 AI 工程师。直接挂载在 GitHub 仓库上，能够从 Issue 直接推导到云端沙箱的代码修改与 Actions 自动化测试，实现端到端无人值守。
        

## 3. VAC 标准流水线流程（Pipeline Framework）

本方案包含**三大标准阶段、六个核心动作**，彻底将研发人员从 Coding 苦力中解放出来，转型为“代码审查员与业务把控者”。

```
[步骤 1: 需求表一键导入] ──> [步骤 2: Lovable 生成可交互原型] ──> [步骤 3: 人类在线修正、固化 Spec]
                                                                        │
                                                                   (自动化同步)
                                                                        ▼
[步骤 6: 云端沙箱自动测试并提 PR] <── [步骤 5: Cloud Agent 认领任务编码] <── [步骤 4: Linear 拓扑任务拆解]
```

### 阶段一：业务减熵与视觉对齐（需求说清楚）

#### 动作 1：表格到可交互原型的瞬间跃迁 (Table-to-Prototype)

用户直接将整张模糊的二级需求表（如：地区发电台账数据上报等 12 个模块）拖拽上传至 **Lovable.dev**。Lovable 调用大模型，不仅做文本分析，还会**在一分钟内原地拔地而起一个完备的前端 Web 系统原型**。

- 表格中模糊提及的“不一致的台账信息并输出台账列表清单”，会在网页上直接变成一个带有高亮红色标记的实时比对表格。
    
- “上报进度条、红色标记”会以真实的动态 UI 组件呈现在网页上，并自带假数据模拟。
    

#### 动作 2：基于真实原型的“所见即所得”澄清 (Socrates Sandbox Dialogue)

人类不再看着枯燥的文档去猜测系统长什么样。直接在 Lovable 的右侧对话框中，对着生成的网页原型向 AI 下达修正指令：

> **人类调整指令：**
> 
> - “地调归档的审核按钮，在县调人员未上报成功前必须处于变灰（Disabled）状态。”
>     
> - “台账属地化管理中，切换县区下拉菜单时，左侧的发电资源数量统计图表（Konva/Echarts）要同步联动刷新。”
>     
>     AI 实时修改，网页原型秒级重构，直到人类确认：“这就是我要的系统”。
>     

#### 动作 3：规格说明书（Spec）与视觉资产冷冻 (Visual Spec Freezing)

当原型确认通过后，Lovable 会自动在后台将该版本的 UI 结构、底层接口 Mock 契约、状态流向以及架构决策（ADR）导出为一份包含高精度图片与代码契约的云端系统说明书（Visual Spec），状态自动冷冻为 `LOCKED`。

### 阶段二：任务切碎与项目编排（中转为 Issue）

#### 动作 4：Linear 拓扑任务链自动化拆解 (Advanced Issue Tokenization)

**Linear** 通过 Webhook 自动吃下这份被锁定的 `Visual Spec`。利用大模型的超长上下文和 Prompt Caching（提示词缓存），以极其低廉的成本（享受公网大模型缓存折扣）将整套系统拆解。

- 拆解不再是盲目的纯文本。Linear 会根据原型的代码结构，精准划分出 `epic:台账管理` 下的 `feature:差异性检查`，并自动计算任务依赖（如：后端提供数据比对 API 的 Task 未完成前，前端渲染组件的 Task 自动处于 `Blocked` 阻塞状态）。
    

#### 动作 5：代码仓库与 Issue 的深度双向互通 (GitLab/GitHub Synced)

任务被自动推送至 GitHub/GitLab Issue。每一个 Issue 内部都精准包含了该原子功能对应的**原型页面截图、API 参数 JSON Schema 以及自动化测试的断言条件（Assertion Criteria）**。

### 阶段三：云端 Agent 自动闭环（代码写明白）

#### 动作 6：云端无人值守编码与沙箱验证 (Cloud Agentic Development)

开发人员或技术负责人甚至不需要打开本地的 VS Code / Cursor。直接在浏览器中打开 **GitHub Copilot Workspace**：

1. **Agent 认领任务：** 云端 Agent（基于顶级大模型）自动认领 Linear 流转过来的 Issue #102（如：编写省地台账差异属性比对算法）。
    
2. **全库感知与高精度编码：** 凭借公网模型强大的海量上下文（200k+）感知力，Agent 瞬时检索整个 GitHub 仓库的代码、Nacos 云端配置文件以及第三方公共依赖库，直接在云端拉取一个临时开发分支（Feature Branch），秒级写完核心比对算法及 JUnit 单元测试。
    
3. **沙箱自动化编译：** Agent 在云端拉起隔离的沙箱环境（E2B / GitHub Actions），自动运行 `mvn clean test` 或 `npm run test`。如果编译报错，Agent 会利用 `Thought-Action-Observation` 循环自己在沙箱内看报错日志改 Bug，直到测试 100% 通过。
    
4. **人类 Review PR 闭环：** 测试通过后，Agent 自动向主分支提起一个 Pull Request (PR)，并附带详细的修改报告。技术负责人在手机或浏览器上确认无误，点击 `Merge`，代码自动通过 CI/CD 部署到 Nacos 和云端环境。
    

## 4. 方案质量红线与合规控制（Quality Gates 3.0）

连接公网并不意味着放弃质量控制，VAC 方案引入了更高级的“契约与运行双重门禁（Runtime Gate）”：

1. **原型契约拦截（Lovable Contract Lock）：** 如果生成的代码接口字段、数据类型与第一阶段Lovable锁定的 Live 原型不匹配，GitHub Actions 自动化 CI 流程将直接打回 PR，绝不允许任何打破业务原型的“技术脑补”发生。
    
2. **云端安全全量审计（SonarCloud + Guardails）：** 每一个由云端 Agent 提起的 PR，在人类审批前，都会强制通过公网的安全扫描工具（如 Snyk, SonarCloud），自动检测代码是否存在内网防爆破漏洞、SQL注入、或者是不合规的依赖包引入。
    
3. **高可用性能 Mock 压测门禁：** 在云端沙箱中，AI 在生成代码的同时，会被强制要求生成一份轻量级压测脚本。在沙箱内模拟 500+ msg/sec 的台账变更吞吐量，如果该比对算法导致 OOM 或 CPU 暴涨，该 Issue 自动被判定为“质量不达标”，退回重写。

## 6. 实施战略与行动指南

1. **原型预对齐第一原则：** 坚决执行“Lovable 原型没对齐，不拆任务、不写代码”的铁律。在前端交互和 API 边界被视觉化锁定前，不向 Linear/GitHub 输送任何 Issue。
    
2. **微调与评论区共识：** 如果在云端 Agent 自动编码过程中需要调整细节，人类无需去改原始表格，直接在该 GitHub Issue 的 `Comments` 评论区输入补充要求（如：“注意字段 capacity 需要支持保留两位小数”），云端 Agent 会自动抓取 Issue 最新评论进行二次迭代编译。
    
3. **算力与成本监控：** 开启 GitHub Actions 和大模型官方 API 的高级权限，合理配置弹性算力组，日常小重构或修 Bug 设定最大 Token 消耗上限，确保项目在极速交付的同时，研发 ROI（投资回报率）达到最优化。
