# 认知架构学与底层存储引擎的边界 (Cognitive Architecture vs Storage Engine)

本篇探讨旨在澄清当前市面上诸多 AI 记忆理论与我们在工程落地（如 QMem）时的边界划分，并深入探究如何借鉴传统计算机科学的底层概念来优化大模型记忆瓶颈。

## 1. 记忆理论的“上下两层”划分

在面对 `ai-memory-wiki` 中海量的缩写与理论（如 SIP 潜意识注入、VAC 等）以及企业招聘中满天飞的“长短期记忆”时，我们需要明确两层定位：

### 上层：认知架构学 (Cognitive Architecture)
- **核心关注**：AI 应当如何“像人类一样思考”？
- **关键词**：长短期记忆 (LTM/STM)、工作记忆 (Working Memory)、非显式存储思想、潜意识注入 (SIP) 等。
- **定位**：这是“大脑设计图”。人类的大脑没有实体关系表，靠的是神经突触权重（非显式）和海马体的工作切片。这一层在研究如何利用模型本身的 Context 窗口与微调，甚至各种玄学的 Prompt 流转来模拟人的意识流。
- **市面代表**：大部分讲“大模型记忆机制”的理论论文，以及 LangGraph 这类负责状态流转的智能体编排框架。

### 下层：工程物理层 (Storage Engine)
- **核心关注**：如何把抽象的经验，以最低成本、最快速度固化到物理介质上？
- **关键词**：SQLite、向量数据库 (sqlite-vec/Milvus)、物理文件隔离 (Markdown Drawer)、触发器硬删除。
- **定位**：这是“实现设计图的物理数据库”。无论认知模型多么宏大，最终的数据必须落盘为 `.db` 记录或 `.md` 文本。QMem 项目 (mojomem) 正是死守在这一层，做极致纯粹的底层引擎。

## 2. 破局猜想：将“C语言指针 / 操作系统注册表”引入 AI 记忆开发

目前 AI 领域面临的最大瓶颈是**“记忆的膨胀”**。将 10 万字的规则全部压入 Prompt，不仅会导致模型崩溃（Lost in the middle），还会造成巨大的 Token 成本浪费。

此时，借鉴传统计算机系统中的**“指针 (Pointer) 和注册表 (Registry)”**概念，显得绝妙且超前：

### 指针 (Pointer) 的本质
指针本身只存储一个内存地址，不存储具体的内容实体。只有在需要用到时，才会去进行**解引用 (Dereference)**。

### 在 AI 记忆中的映射 (Lazy Evaluation of Memory)
在 PRQ (Push-Recall-Query) 的物理隔离方案中，这一概念得到了完美的工程印证：

1. **注册表指向**：
   - 项目根目录的 `CLAUDE.md` 就像是 Windows 的注册表或 C 语言的根指针。
   - 它的底部仅仅包含一句硬连接：`Ref: ../.assets/vue2-common.md`。这仅仅占据了不到 10 个 Token 的体积。
2. **潜意识常驻**：
   - AI 在整个项目周期中，它的全局 Context 里只带有这个指针地址。它知道“有个叫 Vue2 避坑指南的东西存在那个文件里”，但它不需要把内容全部背下来。
3. **显式解引用 (Dereference)**：
   - 当遇到与 Vue2 相关的报错时，大模型会主动调用工具（如 `mem_recall`），传入这个指针地址（文件名）。
   - 系统将物理文件 `.assets/vue2-common.md` 加载入当前对话的短期记忆中。这就完成了一次“地址到实体的懒加载 (Lazy Load)”。

### 总结
这种**“AI 记忆指针化”**的模式，是打破当前所有粗暴 RAG（无脑把 Top-K 文段塞进 Prompt）的降维打击。它不仅大幅降低了上下文污染，还利用了传统计算机科学中最久经考验的“地址与实体分离”哲学。这正是后续我们在探索非显式存储思想时，最值得深挖的工程化方向。
