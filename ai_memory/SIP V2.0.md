# 📋 “系统说明书-Issue”方案（SIP）V2.0 标准实施白皮书

### 暨 基于公网顶尖大模型（GLM-5.2）赋能的超大规模全栈研发流水线

## 1. 方案概述与核心哲学

在企业级私有内网环境的软件研发中，将传统的“需求功能点估算表”直接转化为“可运行代码”存在巨大的技术与管理断层。本方案（Specification-to-Issue Pipeline V2.0, 简称 **SIP V2.0**）是针对“物理内网环境隔离，但后端提供公网顶尖大模型（GLM-5.2）API 及充足额度支持”这一特殊且强力的场景量身定制的工业级研发流水线。

### 1.1 核心痛点

- **信息熵极高（模糊瞎写）：** 原始功能点描述（如电力系统中的“台账比对”、“流程同步”）字数极少、业务边界模糊。若直接交付大模型编码，大模型会严重“脑补”技术方案，导致产出垃圾代码。
    
- **长文本断层与算力白耗：** 传统流水线在面对庞大业务时，频繁重复读取冗长的背景资料，导致首字延迟（TTFT）严重，内网网关高并发下极易崩溃。
    
- **上下文交叉污染：** 复杂的跨级业务（省-地-县）逻辑交织。若让 AI 协同全量上下文，不同模块间极易产生行级语义的交叉污染。
    

### 1.2 SIP V2.0 哲学

引入“系统说明书”**与**“开发 Issue”作为两道物理减熵屏障。全面释放 GLM-5.2 的超长上下文（Long-Context）及大海捞针（Needle in a Haystack）的语义对齐能力，通过工程结构卡死逻辑，通过大模型加速交付。

- **需求说清楚：** 依靠 GLM-5.2 的专家级横向联动能力，将业务模糊性转化为确定性的技术设计方案。
    
- **代码写明白：** 依靠 GitLab Issue 承载原子化任务，利用 Prompt Caching（提示词缓存）让 Claude Code 具备“全库感知”的秒级编码响应。
    

## 2. 全栈工具链选择（Tooling Selection）

基于完全隔离的内网物理限制与公网 API 穿透，方案复用内网已有高价值资产，组装成一条轻量、高可控的 AI 原生代研发工具链：

```
+------------------------------------------------------------------------+
| 阶段一：业务减熵（Dify Workflow / 全量二级需求表导入）                 |
| 动力源：GLM-5.2 (全量深度扫描、专家级横向追问、ADR自动草拟)           |
+------------------------------------------------------------------------+
                                   │ (生成：结构化说明书 + 状态元数据)
                                   ▼
+------------------------------------------------------------------------+
| 阶段二：任务切碎（LangGraph / 内网 GitLab REST API 灌入）               |
| 动力源：GLM-5.2 (红蓝对抗评审、按组件/模块进行有向无环图拆解)           |
+------------------------------------------------------------------------+
                                   │ (生成：模块化/组件级开发 Issues)
                                   ▼
+------------------------------------------------------------------------+
| 阶段三：精准编码（Claude Code / 挂载 GitLab MCP Server）                |
| 动力源：GLM-5.2 + Prompt Cache (说明书缓存、多源上下文全量感知)         |
+------------------------------------------------------------------------+
```

- **业务减熵层：Dify（开源自托管版）**
    
    - _选型理由：_ 支持完全离线 Docker 部署。利用其强大的 Workflow 节点、迭代器与表单交互能力，作为扮演高级业务分析师（BA）的载体。本次 V2.0 升级直接将全量二级需求表一键导入，利用 GLM-5.2 进行整表交叉语义分析。
        
- **任务编排层：LangGraph + GitLab / Jira API**
    
    - _选型理由：_ 利用有向无环图（DAG）的严谨状态控制。将通过评审的说明书，按照预设的标准 JSON Schema 自动切碎成模块化任务，通过 REST API 灌入内网 GitLab/Jira。
        
- **精准执行层：Claude Code + GitLab MCP Server**
    
    - _选型理由：_ 研发人员本地主力工具。通过离线挂载 GitLab MCP 插件，赋予 Claude Code 联动内网 Issue 的能力。在 V2.0 中，彻底释放 GLM-5.2 的超大上下文，允许 Claude Code 在编码时携带全量说明书与代码骨架。
        

## 3. SIP V2.0 标准流水线流程（Pipeline Framework）

本方案包含**三大标准阶段、六个核心动作**，严格遵循 **Maker-Checker（四眼原则）** 安全红线。

### 阶段一：业务减熵（需求说清楚）

#### 动作 1：全量需求透传与跨模块扫描 (Bulk Ingestion & Global Scan)

用户将整张二级需求表（包含地区发电台账上报、台账差异性检查、台账模板管理、台账映射管理等所有行项目）一次性导入 Dify 系统的输入端。GLM-5.2 启动全局上下文扫描，它不会割裂地看待单行需求，而是进行**横向联动分析**（例如：发现“台账上报管理”中的组装逻辑与“台账映射管理”中的数据列匹配具有强强依赖关系）。系统自动将有歧义或依赖缺失的模块打上 `NEED_CLARIFY`（待澄清）元数据标签。

#### 动作 2：专家级一针见血式澄清 (Expert-Level Dialogue)

AI 拒绝机械式、挤牙膏式的连环追问，而是利用强大的行业常识，整理出一份精简、高密度的《高级架构对齐清单》，每次抛出 3-5 个直接决定架构走向的核心抉择。

> **电力系统实操示例：**
> 
> - **AI 追问：** “在‘省地流程同步’和‘运行状态变更跟踪’模块中，均提到了本侧与省调的数据比对。请明确：本系统与省公司系统的通信通道是基于**内网 MQ 消息总线（如 Kafka）的实时异步监听**，还是**定时任务调用省调 REST API**？这直接决定了差异比对服务是采用‘行级实时触发’还是‘批量定时任务’架构。”
>     

#### 动作 3：固化系统说明书与 ADR (Specification & ADR Freezing)

人类（架构师/PO）针对关键技术通道做出答复后，GLM-5.2 将这些共识固化，自动输出一份标准的 Markdown 系统说明书。

该说明书必须强制包含 **ADR（架构决策记录）** 章节。例如：

Markdown

```
## ADR-003: 省地两侧电源状态同步方案选择
* **上下文：** 运行状态变更后需自动同步省地两侧电源状态。系统需支持 500+ msg/sec 级别的并发台账状态变更。
* **决策：** 鉴于内网基础设施完善，选择【方案B：引入分布式定时任务（Xxl-Job）+ 延迟队列（RocketMQ）】，凌晨拉取全量，白天通过延迟队列监听增量。利用内网 Nacos 动态配置调度参数。
* **当前状态：** [WAITING_FOR_HUMAN_CONFIRM] <-- 等待架构师一键确认
```

人类确认后，说明书状态变更为 `APPROVED`。

### 阶段二：任务切碎（中转为 Issue）

#### 动作 4：结构化组件化拆解 (Modular Tokenization)

编排引擎（LangGraph）读取 `APPROVED` 的系统说明书。此时，利用 GLM-5.2 的大文本能力，启动两个并行的虚拟 Agent 角色（BA 智能体与 QA 智能体）进行内部的“红蓝对抗评审”，自动推导出边界异常流。评审通过后，系统严格按照“数据库 DDL $\rightarrow$ 核心算法组件 $\rightarrow$ 后端 API 控制器 $\rightarrow$ 前端视图渲染”的组件级依赖链条切碎需求，生成标准 JSON 任务流。

#### 动作 5：内网研发生态灌入 (Issue Rest-Injection)

流水线调用内网 GitLab/Jira 的 REST API，将 JSON 任务流批量灌入指定的 Git 仓库中变成物理 Issue。Issue 的标题、描述、Acceptance Criteria（验收标准）天然成为了该任务的分布式状态机，且 Issue 的评论区（Comments）将作为人类开发阶段微调需求的唯一事实源。

### 阶段三：精准编码（代码写明白）

#### 动作 6：多源上下文全量感知编码 (Context-Rich Atomic Coding)

开发人员在本地激活 Claude Code，通过 MCP 协议拉取指定的 `Issue ID`（如：#102 编写省地台账差异属性比对算法）。

- **Prompt Caching（提示词缓存）爆发：** 此时，Claude Code 会把**整份庞大的系统说明书、现有的 Nacos 配置文件、已有的基础 Entity 类以及当前的 Issue 描述**作为全量上下文打包发给 GLM-5.2。
    
- 因为大模型额度足够且支持 Prompt Cache，这份重复读取的高达数十万字的基础说明书和框架代码将 100% 命中大模型网关缓存。首字延迟（TTFT）直接进入毫秒级，且算力成本与响应延迟暴跌。
    
- GLM-5.2 凭借其高超的“大海捞针”能力，在超大上下文里精准感知到内网 Maven 私服里其他团队已经写好的标准电力工具包，写出极具复用性、包含完备 JUnit 单元测试的高质量代码。代码编译通过后，自动执行 `git commit -m "feat: close #102"` 触发内网 CI 流水线。
    

## 4. 说明书完整性控制（Quality Gates）

为了确保从需求表到代码的转化质量，SIP V2.0 方案在阶段一和阶段二之间强制设立了**三大自动化质量红线（Linting 机制）**：

1. **Schema 强结构校验：** 说明书必须采用标准的 Front Matter 声明状态。必须强制包含 `## 核心数据模型（ILF/EIF映射）`、`## 安全与权限边界（如：县调属地化行级过滤隔离）`、`## 异常处置策略（如：自动纠错失败后的熔断机制）` 章节。任何一章缺失，系统自动拦截，拒绝向 GitLab 拆解 Issue。
    
2. **ADR 唯一性锁定：** 凡是涉及多种实现方案的通用功能（例如：台账模板检索、流程同步撤销），说明书中必须存在对应的 `ADR-XXX` 记录，列出至少两种备选方案的优缺点，且状态必须为人类显式确认的 `[CONFIRMED_BY_HUMAN]`。
    
3. **异常流红蓝对抗（Red Teaming）：** 评审 Agent 会拿着“内网高可用和合规检查单”对说明书进行自动化拷问。例如：“说明书提到‘自动根据模板组装上报数据’，请问如果本地台账数列与模板数列不匹配，字段映射发生断裂（Null），系统是否会抛出空指针异常？异常处置策略是什么？”生成 Agent 必须在说明书中补齐该 Exception Handling 逻辑，质量红线才予放行。
    

## 5. SIP V2.0 落地成效与优势对比

得益于公网顶尖大模型（GLM-5.2）与充足额度的强力支撑，SIP V2.0 方案相比于传统的内网小模型流水线实现了跃迁：

|**评估维度**|**传统内网方案（基于局域网小模型）**|**SIP V2.0 方案（GLM-5.2 赋能版）**|
|---|---|---|
|**输入吞吐量**|分批、零散输入，容易顾此失彼|**一次性全量导入整张二级需求表（12个模块同步分析）**|
|**人机交互频次**|机械式、挤牙膏式，人类需要回答大量细碎问题|**专家级、跨模块联动、一针见血，交互次数缩减 70%**|
|**说明书信息密度**|刻意控制说明书体积，防止模型理解崩溃|**内容极其详实，允许包含海量 ADR、Edge Cases 与时序图**|
|**任务拆解颗粒度**|必须切得极碎（原子化），导致 Issue 数量泛滥|**模块化（按组件/服务），留给大模型合理的发挥空间**|
|**Claude 编码感知**|严格限制“单 Issue 盲盒编码”，AI 看不到项目全貌|**全上下文感知（说明书 + 全库代码 + Nacos配置），代码复用率极高**|
|**响应速度 (Latency)**|高并发下内网大模型频繁触发 429 挂起，延迟严重|**100% 命中 Prompt Cache，秒级返回，首字响应达毫秒级**|

## 6. 实施规范与维护指南

1. **状态机维护：** 严禁跳过阶段一直接手动去 GitLab 创建 Issue。所有有效的开发 Issue 必须带有 `SIP-Generated` 标签，确保其唯一事实源可追溯至系统说明书。
    
2. **日志规范：** 负责中转的 Python/TypeScript 脚本在调用 GitLab API 及 GLM-5.2 API 时，所有 Debug 日志必须重定向至本地 `logs/sip_pipeline.log`，严禁污染标准输出（`stdout`），以防破坏 Claude Code 的 MCP 协议 JSON-RPC 流。
    
3. **缓存管理：** 当系统说明书发生重大版本变更（如业务逻辑重构）时，需在 Dify 端手动触发一次测试调用，以提前“预热”（Warm-up）GLM-5.2 的云端 Prompt Cache，确保后续开发人员开工时能直接享受到高并发、低延迟的编码体验。
