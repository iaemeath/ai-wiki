# 📐 “多维视觉骨架与双轨业务减熵流水线”方案（WDD）标准实施白皮书

### 暨 内网隔离环境下基于隔离单文件（SFC）视觉催化与硬核业务收敛的敏捷研发框架

---

## 一、 方案概述与演进哲学

在企业级私有内网环境（如电力、金融等调度与中台系统）中，需求的转化面临着双重挑战：文字描述过于抽象，导致人类与 AI 都在“脑补”画面；而直接构建复杂的端到端高保真系统，在内网又会遭遇生成速度慢、网络不通、代码耦合等工程硬伤。

### 1.1 核心痛点

* **文字说明书的“冷冰冰幻觉”：** 纯文字或 Markdown 形式的系统说明书虽然逻辑严密，但在界面布局、交互联动等维度极度缺乏直观性，往往在系统开发过半时，人类才发现前端交互不符合预期。
* **重型原型系统在内网的“水土不服”：** 试图在内网复制公网 VAC 方案（Lovable/V0）的复杂全套应用生成，需要改动微前端基座、路由、甚至是后端 Nacos 配置。这导致第一阶段耗时过长，且多模块间极易产生代码交叉污染。
* **“画页面”与“理业务”的本末倒置：** 很多时候研发团队花费大量精力去写完美的前端代码，却忽略了**底层的核心业务逻辑和算法边界（如省地数据通信通道、异常纠错策略）才是系统的灵魂**。

### 1.2 WDD 方案哲学

WDD 方案的核心哲学是：**“以隔离视觉骨架为催化剂，用最快的速度锁死业务说明书”**。

* **第一轨（多维隔离视觉骨架）：** 页面不是最终的生产代码，它的存在**纯粹是为了“看”和“辅助直观感受”**。必须生成极快、物理彻底隔离、井水不犯河水。
* **第二轨（关键业务说明书）：** 网页只是外壳，业务才是核心。利用看页面时被激发出的人类思维漏洞，反向压榨出高密度的关键业务说明书（ADR、业务场景、算法逻辑）。

## 二、 全栈工具链选择（Tooling Selection）

WDD 方案在内网环境下追求极致的轻量与效率，工具链的设计核心是“轻量级本地运行时沙箱”：

* **业务减熵与代码直出层：Dify + GLM-5.2 (公网穿透接口)**
* *选型理由：* 利用自托管的 Dify 导入全量二级需求表。由 GLM-5.2 依托其强大的长文本推理能力与代码高频直出能力，跳过复杂的架构分层，**直接将各个功能模块吐成“面条式”的纯前端单文件组件代码**。


* **秒级视觉催化层：内网离线沙箱 / OpenWebUI (Artifacts 机制)**
* *选型理由：* 部署在内网的轻量级 HTML/Vue 动态渲染运行时（如基于 `vue-playground` 离线版）。它不与任何内网正式数据库、网关连接，只负责将大模型吐出的隔离单文件代码**在浏览器右侧秒级渲染成可以点击、高亮的视觉骨架**。


* **核心任务承载层：内网 GitLab Issue + 关键业务说明书**
* *选型理由：* 锁定的“双轨交付物”自动同步到 GitLab，作为后续 Claude Code 精准开发的权威底稿。

## 三、 WDD 标准流水线流程（Pipeline Framework）

WDD 方案包含**三大阶段、五个核心动作**，将“看图对齐”与“硬核理业务”完美结合。

```
[步骤 1: 全量二级需求表导入] ────> [步骤 2: GLM-5.2 秒级生成 12 个物理隔离的 Mock 页面 (.vue)]
                                                              │
                                                              ▼
[步骤 4: 固化双轨资产：业务说明书 + 隔离骨架] <── [步骤 3: 人类点击 Mock 页面，反向排查业务漏洞并补全方案]
                      │
                 (自动化同步)
                      ▼
[步骤 5: GitLab 任务拆解] ────> [Claude Code 进场：严格参考 WDD 双轨资产，编写生产级后端与前端代码]

```

### 阶段一：多维视觉催化（生成要快、要隔离）

#### 动作 1：物理隔离的面条式 Mock 页面秒级生成 (Sandboxed SFC Generation)

用户将整张二级需求表（12个电力台账模块）一键导入。系统不进行任何后端架构分层设计，而是命令 GLM-5.2 针对每一个行项目，在应用模板的 `src/mock-views/` 目录下，生成 12 个**彼此完全独立、没有任何路由依赖的单文件组件（Single File Component, .vue 或 .tsx）**。

* **机制：** 每个文件都是“面条代码”，即：HTML 结构、Vue 响应式变量、内网标准的 UI 样式、以及模拟的本地 JSON 假数据（Mock Data）**全死死写在一个文件里**。
* **优势：** 这种页面没有任何外部依赖耦合。AI 修改模块 A（台账差异性检查）时，绝对不可能改垮模块 B（省地流程同步），生成和重构速度达到**秒级级响应**。

#### 动作 2：基于隔离骨架的业务缺陷反向排查 (Socrates Screen Review)

开发负责人打开内网离线沙箱浏览器，肉眼审查这 12 个隔离的骨架页面。在真实的“看页面”过程中，由于视觉组件的直观刺激，人类的架构师思维会被瞬间激活，开始**反向捕捉原本被文字掩盖的业务漏洞**：

> **人类基于视觉原型的深度思考：**
> * “我看到 AI 在‘省地流程同步’页面上画了一个撤销按钮。那么，如果省调那边由于网络原因导致撤销操作不成功，这个 Mock 页面上给出的‘失败提醒信息’是应该弹窗、还是写入历史日志？如果撤销成功了，本地台账的状态应该自动变更为什么值？”
> 
>

### 阶段二：关键业务减熵（理业务、出 Spec）

#### 动作 3：双轨事实源冻结 (Dual-Track Spec Freezing)

人类将这些由视觉骨架逼出来的、真正决定系统灵魂的业务核心方案（如高可用防重机制、Nacos 配置项、数据流转断言）输入给 GLM-5.2。
大模型立刻进行处理，产出“双轨事实源”并一键冻结：

1. **第一轨（多维隔离视觉骨架）：** 被锁定的、包含基础 UI 元素的静态 `.vue` 文件。
2. **第二轨（关键业务说明书）：** 明确写死核心业务方案、异常流策略与 ADR 决策的 Markdown 文本。

#### 动作 4：GitLab 敏捷任务映射 (GitLab Issue Mapping)

流水线读取双轨资产，在内网 GitLab 自动生成开发 Issue 链。不同于 SIP 的盲盒文字，这里的每个 Issue 内部都直接挂载了**对应隔离 Mock 页面的源码和截图**，同时强绑定了**关键业务说明书里的算法逻辑章节**。

---

### 阶段三：真实编码落地（代码写明白）

#### 动作 5：带图纸的高精度真实编码 (Blueprint-Driven Coding)

本地开发人员唤醒 **Claude Code**，挂载刚才锁定的双轨资产开工：

* **在业务层：** Claude Code 读取《关键业务说明书》，在 `src/main/java/` 下极其严谨地编写 Spring Boot Controller、Service、分布式锁、自动纠错的线程池以及配置 Nacos 控制台参数。
* **在页面层：** 真实编写前端代码时，前端工程师或 Claude Code 拿第一阶段生成的隔离 Mock 文件当作“动态高精度设计图纸”。直接参考里面的组件拼装和假数据逻辑，将其拆解并正确重构成符合公司生产级路由、Vuex/Pinia 状态管理、以及网关路由的正式生产级前端代码。

## 四、 WDD 方案的自动化质量红线（Quality Gates）

为确保 WDD 方案中“纯看的页面”能真正反向催化出“完美的业务说明书”，方案强设立了两道质量卡口：

1. **业务说明书“空值拦截门禁”（Null-Spec Gate）：**
如果大模型生成的隔离 Mock 页面中包含任何带有“变更”、“提交”、“撤销”等写操作的按钮，那么《关键业务说明书》中**必须强行存在对应的 `ADR` 记录或异常流处置逻辑**。如果发现页面有按钮但说明书没有写逻辑方案，红线判定为 `INCOMPLETE`，拒绝拆解 Issue。
2. **组件契约级覆盖检查（Component Coverage Lint）：**
系统会自动 Lint 大模型吐出的隔离 Vue 文件，确保其调用的组件必须 100% 属于公司内网私服发布的标准中台组件库。任何脑补的、不存在的私有组件标签将直接引发流水线编译报警。

## 五、 三大方案终极横向对比：SIP vs VAC vs WDD

| 评估维度 | SIP 方案（内网冷文本版） | VAC 方案（公网全托管版） | **WDD 方案（内网多维双轨版）** |
| --- | --- | --- | --- |
| **第一阶段核心产物** | 标准 Markdown 关键业务说明书 | Lovable 生成的在线 Live App 原型 | **隔離单文件骨架（Vue/React）+ 业务说明书** |
| **第一阶段输出速度** | 慢（需要大模型用文字极其细碎地构思） | 快（依赖公网成熟 SaaS 的全自动云端构建） | **极快（秒级，大模型直出互不干扰的独立面条代码）** |
| **人类审查体验** | 读密密麻麻的技术文字，难以形成直观体感 | 在 SaaS 网页上直接点击，体验极佳 | **在内网浏览器里看隔离骨架，用视觉刺激排查业务漏洞** |
| **模块隔离度** | 一般（上下文容易在多轮对话中交叉污染） | 极高（云端完全托管隔离） | **绝对隔离（12个模块切成12个独立文件，井水不犯河水）** |
| **核心战略定位** | 试图通过严格的文档步骤在内网控死逻辑 | 试图通过公网先进 SaaS 链实现全自动少人化交付 | **在内网环境下，用最轻量的隔离视觉骨架做诱饵，榨出系统的核心业务说明书** |
| **最终编码执行** | Claude Code 基于纯文字盲盒编写全栈代码 | 云端 Agent（GitHub Workspace）自动提 PR | **Claude Code 严格将隔离页面当成参考图纸，把业务和真实代码写明白** |

## 六、 实施规范与行动指南

1. **绝不纠结第一轨的代码质量：** 严禁第一阶段的架构师去纠正隔离 `.vue` 文件里的代码是否优雅、是否分层。记住，第一轨资产只是“看样板、锁需求”的工具，它唯一的指标就是生成快、样式对。
2. **业务说明书的至高事实源：** 一旦开始第三阶段开发，所有涉及核心算法、Nacos 参数以及异常处理的问题，必须以第二轨的《关键业务说明书》为最高准则，隔离 Vue 文件只提供视觉参考。
3. **大模型上下文预热：** 依托内网 GLM-5.2 的强大额度，在导入整张二级需求表时，必须在 Dify 提示词中加入公司标准的中台 UI 组件库（Element Plus/AntD等）的完整 DTD 或 Schema 规范片段，确保大模型生成的隔离面条代码在视觉上与公司正式系统达成 100% 的视觉一致性。
