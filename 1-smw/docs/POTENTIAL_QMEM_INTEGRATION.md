# 1-SMW 与 QMem (mojomem) 融合前景探讨

> **状态**: 当前为独立项目。目前 1-SMW 作为顶层业务编排应用（基于 LangGraph），QMem 作为底层极速记忆引擎（基于 Mojo/SQLite），两者完全独立。本记录仅用于未来技术演进的备忘。

## 1. 架构定位的天然互补
- **1-SMW (业务大脑)**: 专注于需求工程流转、状态机维护、苏格拉底追问逻辑，以及“Human-in-the-loop”的交互编排。它需要知道“当前该做什么”。
- **QMem (外接海马体)**: 专注于历史经验的持久化、知识图谱与向量语义的毫秒级“推/拉/查”。它能提供“我们过去是怎么做的”、“我们的环境红线是什么”。

## 2. 潜在的融合场景 (Use Cases)

### 场景 A：需求工程的“历史经验防抖”
在 1-SMW 的工作流中，AI 分析 Excel 需求时，可能会忽略掉旧架构的某些硬性约束。
- **融合路径**：让 1-SMW 的 LangGraph 在 `analyze_requirements` 节点触发时，通过系统调用自动调用 QMem 的 `mem_recall`。将用户的需求作为 query，查询 QMem 库中是否有相关的“历史架构教训”或“技术红线”。
- **收益**：AI 产出的 ADR 和 Mermaid 图将严格遵循公司的既往共识（Q2），避免重复踩坑。

### 场景 B：ADR 自动沉淀为记忆 (Knowledge Consolidation)
1-SMW 每次产出的 ADR（架构决策记录）本质上就是最高质量的“项目共识”。
- **融合路径**：在 1-SMW 工作流的终点（生成输出物时），触发一个回调钩子，调用 QMem 的 `mem_save`（带上特定的 Topic Key，如 `ADR_OUTPUT`），将这次生成的决策摘要存入 `core_memory.db` 并提权。
- **收益**：形成真正的记忆飞轮。今天的输出，自动成为明天的上下文。

## 3. 合并形式建议
由于两者底层语言（Python vs Mojo）和哲学不同，若未来发生整合，不建议进行源码级的物理合并，而是**保持微服务化架构**：
- QMem 以独立 MCP 服务或本地微服务后台驻留。
- 1-SMW 仅在 LangGraph 的 Tool Node 中注册 QMem 的 API/CLI 客户端进行网络或进程间调用。保持“解耦且协同”。
